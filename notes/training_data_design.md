# Training Data Design — Sprint 2

This document captures the decisions behind the Sprint 2 training-data
construction and the artifacts it produces. The pair file is regenerated by
`scripts/build_training_pairs.py`; the hard-negatives file by a script added
later in this sprint. Both live under `data/training/` and are gitignored
(too large to commit; reproducible from these scripts).

## Held-out enforcement

The held-out set is derived **directly from the eval dataset**, not from
`results/check1_leakage.json`. We exclude **all opinions referenced by any
relevance judgment** — score 0, 1, or 2 — yielding 624 opinions out of 14,094
unique (4.43%). This is more conservative than the 596 positive-only count
in the leakage report by 28 opinions; the cost in training-data volume is
negligible and the rule is simpler to defend ("anything a human looked at
during eval construction is off-limits for training").

The pair builder enforces this at row-emit time and emits a count of
excluded held-out opinions in the coverage report.

## Pair-file schema

One row per `(question, opinion)` pair where the opinion is not held out and
has at least one usable question text (real or synthetic, post-normalization).

| Field | Type | Meaning |
|---|---|---|
| `opinion_id` | string | Canonical opinion ID. |
| `year` | int | Year of opinion. |
| `question` | string | Normalized query text — see "Question normalization" below. |
| `question_source` | `"real"` \| `"synthetic"` | Provenance of the query text. |
| `qa_source` | `"extracted"` \| `"synthetic"` \| `"mixed"` | Corpus's own provenance flag for how `qa_text` was assembled. |
| `topic_primary` | string \| null | Corpus-assigned topic. |
| `pos_qa_text` | string | `embedding.qa_text` — the corpus executive summary. |
| `pos_body` | string \| null | `facts + analysis + conclusion`, joined with `\n\n`. **Null** when the opinion lacks a real `sections.analysis` (rather than fall back to noise). |
| `pos_qa_plus_body` | string \| null | `qa_text + "\n\n" + body`. Null whenever `pos_body` is null. |
| `pos_full_text` | string | `content.full_text` — entire opinion. Always present. |
| `has_real_question` | bool | Real `sections.question` was non-empty. |
| `has_synth_question` | bool | `sections.question_synthetic` was non-empty. |
| `has_facts`, `has_analysis`, `has_conclusion` | bool | Per-subsection coverage flags. |
| `body_complete` | bool | True when `pos_body` is non-null (analysis present and ≥ 1 of facts/conclusion). |

### Question source preference

When both `sections.question` (real) and `sections.question_synthetic` exist
(1,684 opinions in the corpus), we use the real one. Rationale: the synthetic
was generated *because* the corpus author wanted a backup for noisy extraction;
when the real extraction succeeded, the synthetic is redundant. A future
ablation could explore preferring synthetic for opinions where real
extraction quality is low (e.g., high OCR-artifact ratio), but v1 keeps the
selection deterministic.

### Question normalization

Raw `sections.question` text often carries OCR/extraction artifacts like
leading `"1. "`, `"QUESTION:"`, `"Issue —"`, multiple section numbers, and
collapsed whitespace. We strip:

- Up to two passes of leading numeric list markers (`1.`, `(2)`, etc.).
- Up to two passes of leading label prefixes (`QUESTION:`, `QUESTIONS PRESENTED:`,
  `ISSUE:`, etc., case-insensitive).
- Runs of whitespace → single space.

Then we drop rows where the normalized question is shorter than 15 characters
(pathological extractions like `"?"`).

### Positive-doc columns rationale

Four columns per row so Sprint 3 can ablate which `(query → positive_doc)`
representation produces the strongest fine-tune. The columns correspond to
distinct hypotheses about what the model should encode:

- **`pos_qa_text`** — what today's production index uses. Concentrates
  the topic signal at the head of the document, which dense encoders attend
  to most ("Dwell in the Beginning"; Coelho et al. 2024). The default
  baseline.
- **`pos_body`** — the IRAC reasoning (facts → analysis → conclusion), which
  includes the statutory citations that drive the hybrid's
  `CitationScoreFusion` path in Experiment 009. Excludes the executive-summary
  question text to test whether the model learns better from grounded
  reasoning vs from the FPPC's question paraphrase.
- **`pos_qa_plus_body`** — concatenation of both. Head-biased toward the
  executive summary while still carrying the statutory-grounding signal.
- **`pos_full_text`** — the entire opinion. Only meaningfully different from
  the above when sections extraction was partial (older opinions). Acts as a
  fallback that always has content.

The 8192-token native context of the chosen fine-tune target
(`Snowflake/snowflake-arctic-embed-l-v2.0`) means all four columns are
viable without truncation up to p99 of corpus length. The longest tail
opinions (max ~120 KB qa_text, ~170 KB body, ~70 KB full_text) will still get
truncated at model max_seq during training; we accept this for the <1% of
rows it affects.

### Fallback semantics

- `pos_qa_text`: always present (corpus guarantees `embedding.qa_text`).
- `pos_body`: **null** when `sections.analysis` is absent (~31.5% of pairs).
  The trainer should filter these rows out for body-trained variants rather
  than fall back to a noisier representation, so we don't conflate "no body
  content" with "body content of value." Synthetic conclusions
  (`conclusion_synthetic`) ARE used as a fallback for the conclusion subsection
  inside `pos_body`, since they're explicitly LLM-generated to fill that gap.
- `pos_qa_plus_body`: null whenever `pos_body` is null.
- `pos_full_text`: always present.

## Validation slice

Built by `scripts/build_val_slice.py` from `pairs.jsonl`. Targets 5% of pairs
(actual: **543 of 10,806 = 5.02%**), stratified by year (each year contributes
`round(0.05 * year_count)` rows). Seed = `20260521` for reproducibility.

The slice writes full-schema rows (same columns as `pairs.jsonl`), so the
trainer can validate using the same positive-doc columns as it trains on. The
trainer is expected to read `pairs.jsonl` and skip any `opinion_id` present
in `val_slice.jsonl`.

Distribution check vs population (largest deltas):

| Field | Val % | Pop % | Δ |
|---|---:|---:|---:|
| question_source = real | 88.95 | 89.00 | -0.05 |
| question_source = synthetic | 11.05 | 11.00 | +0.05 |
| body_complete = True | 68.88 | 68.54 | +0.34 |
| topic = conflicts_of_interest | 59.48 | 56.49 | +2.99 |
| topic = campaign_finance | 15.47 | 18.50 | -3.03 |
| topic = (unclassified) | 11.23 | 11.18 | +0.05 |

Largest deviation is campaign_finance underrepresentation (~3pp), which is
sampling noise at n=543 and not a stratification failure. COI is slightly
over-represented (+3pp), which is fine since COI is the explicit subgroup
gate. We did not stratify on topic because year-stratification preserves
topic balance closely enough and adds no implementation complexity.

Coverage report: `data/training/val_slice_coverage.json`.

## Hard negatives

Built by `scripts/mine_hard_negatives.py`. For each `opinion_id` in the
training pool (= `pairs.jsonl` minus `val_slice.jsonl`, 10,263 rows) we mine
two sources of hard negatives:

1. **BM25 top-k** against the corpus, querying with the *training question*
   text. Take the top results that are *not* the positive opinion. Reuses
   the search-lab's BM25 engine
   (`/home/nick/Projects/fppc-opinions-search-lab/src/engines/bm25_full_text.py`
   — class `BM25FullText`), so the negatives reflect what the production
   retriever actually surfaces. Configured to keep up to 5 per row after
   over-fetching 30 candidates (the over-fetch buffer absorbs exclusions).
2. **Same-statute different opinion**: any other corpus opinion whose
   `citations.government_code` set overlaps with the positive's, after
   normalizing subsection markers (`87103(a)` → `87103`). Candidates are
   ranked by **shared-code count** so we prefer opinions that overlap on
   many statutes rather than a single generic one (e.g., bare `1090`).
   Up to 5 per row.

Constraints applied to every candidate pool:
- The positive opinion itself.
- The 624 eval-referenced held-out opinions
  (same `load_held_out()` logic as `build_training_pairs.py`).
- The 543 val-slice opinion IDs (mild leakage to avoid: don't let the
  trainer see val opinions as negatives at train time).

Dedup: when an opinion appears in both source pools, we keep one record
and merge — `source = "bm25+same_statute"`, both the BM25 rank/score and
the shared statute list are retained.

### Output format

`data/training/hard_negatives.jsonl`, one line per training opinion:

```json
{
  "opinion_id": "A-19-008",
  "negatives": [
    {"opinion_id": "16-117",  "source": "bm25",         "rank": 1, "score": 12.34},
    {"opinion_id": "11-002",  "source": "same_statute", "shared_codes": ["1090", "87100"]},
    {"opinion_id": "08-219",  "source": "bm25+same_statute",
     "rank": 3, "score": 8.21, "shared_codes": ["1090"]}
  ]
}
```

The trainer joins this with `pairs.jsonl` on the fly. `pos_full_text`
(or `pos_qa_text`, depending on the experiment) of each negative
`opinion_id` is used as the negative document.

### Cost

CPU-only, single-process. Wall time: 2,485 seconds (~41 min) on Strix Halo
for 10,263 rows = ~4.1 rows/sec. The statute graph is built in <1 sec; the
bottleneck is BM25 scoring (each query scores against all 14,096 docs).
Reuses the cached BM25 index at
`../fppc-opinions-search-lab/indexes/BM25FullText_index.pkl`.

## Known limitations / future improvements

- **OCR noise in pre-1985 questions** is real ("Hayor"/"Co~~ittee" etc.).
  Modern opinions are clean. The model will see noisy queries paired with
  noisy docs in training, and clean queries paired with same-noisy docs at
  inference. Worth a future ablation: filter or down-weight OCR-noisy pairs
  by detecting common-substitution rates, or prefer synthetic questions for
  opinions where the real-question extraction is below an OCR-quality
  threshold.
- **`question_source` preference is naïve**: we always use real when both
  exist. For opinions where synthetic is cleaner than noisy-real, we may be
  using lower-quality queries. A Sprint 3 ablation could test: real-only,
  always-prefer-synthetic-if-exists, and a quality-gated mix.
- **Topic field is missing for 11.2% of pairs.** Not currently used in
  training but would matter if we stratified by topic later. The corpus's
  `classification.topic_primary` is the source — could be backfilled with a
  light classifier if needed.
- **One duplicate opinion ID** across the corpus's year folders (14,096 files
  → 14,094 unique IDs). The script silently dedups by first-seen; this is
  fine but worth noting if anyone investigates the count discrepancy.
